## Decisions Made
- FormData POST never sets Content-Type manually — browser sets multipart boundary automatically; plan spec explicitly calls this out
- Paste handler scoped to document (not shadowRoot) — shadow roots do not receive paste events when focus is outside the shadow tree; global document listener required
- autoScreenshot file named 'screenshot-auto.jpg', attached images named 'screenshot-{i}.png' — consistent convention for backend endpoint to parse

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Paste handler moved from shadowRoot to document**
- **Found during:** Task 2 — human browser verification
- **Issue:** Paste (Ctrl+V) was not firing when the widget form was focused because shadowRoot does not receive paste events propagated from document; the event listener was attached to shadowRoot.host instead of document
- **Fix:** Moved ClipboardEvent paste listener registration from `shadowRoot` to `document` in upload.ts / widget.ts paste wiring
- **Files modified:** widget/src/widget.ts (paste listener registration)
- **Verification:** Human confirmed Ctrl+V paste of an image is listed as attached in the form before submit
- **Committed in:** 7decec6 (fix commit)

## Issues Encountered

Paste handler scope mismatch surfaced only during real browser testing (Ctrl+V in form had no effect). Root cause: Shadow DOM does not bubble ClipboardEvent to the shadow root; the listener must be on document. Fixed immediately per Rule 1 and committed as 7decec6.
