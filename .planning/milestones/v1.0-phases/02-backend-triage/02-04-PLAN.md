  <action>
Create `backend/railway.toml` with the following configuration:

```toml
[build]
builder = "NIXPACKS"
buildCommand = "npm install"

[deploy]
startCommand = "npm start"
healthcheckPath = "/health"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 10
```

Field explanations:
- `builder = "NIXPACKS"` — Railway's Node.js auto-detection builder. Detects Node.js from package.json.
- `buildCommand = "npm install"` — Installs dependencies. No separate build step needed because the start command uses tsx for TypeScript execution at runtime.
- `startCommand = "npm start"` — Maps to `node --import tsx src/index.ts` from package.json scripts.start. This is the production entry point.
- `healthcheckPath = "/health"` — Points to the health endpoint in `backend/src/routes/health.ts` which returns 200 `{ status: "ok" }`. Railway checks this path after deployment to confirm the service is ready.
- `healthcheckTimeout = 300` — 300 seconds (5 minutes) for the health check to succeed. Default Railway timeout; sufficient for npm install + server startup.
- `restartPolicyType = "ON_FAILURE"` — Automatically restarts the service if it exits with a non-zero exit code. This ensures always-on behavior per BACK-07. Use ON_FAILURE (not ALWAYS) to avoid restart loops on clean shutdown.
- `restartPolicyMaxRetries = 10` — Limits restart attempts to prevent infinite crash loops.

Do NOT use `ALWAYS` restart policy — `ON_FAILURE` is correct because:
1. It restarts on crashes (non-zero exit) ensuring always-on behavior
2. It does not restart on clean `SIGTERM` shutdowns during redeployment
3. Railway documentation recommends ON_FAILURE for web services
